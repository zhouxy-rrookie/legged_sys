# `legged_sys` 仓库 AGENTS 指南

## 适用范围
- 本文件适用于仓库根目录 `/mnt/d/legged/legged_sys` 下的全部内容。
- 实际固件工程位于 `legged/` 目录。
- 这是一个基于 STM32H723 的裸机/RTOS 固件项目，使用 STM32CubeMX 生成基础代码，并通过 Keil MDK-ARM 工程进行构建。

## 仓库结构
- `legged/Core/`：STM32Cube 生成的入口、HAL 初始化、RTOS 启动和中断处理。
- `legged/algorithm/`：项目自定义算法代码，包括运动学、PID、轨迹等。
- `legged/driver/`：电机协议与设备通信相关代码。
- `legged/bsp/`：板级支持封装。
- `legged/USB_DEVICE/`：STM32 USB CDC 设备栈适配代码。
- `legged/Drivers/`：STM32 HAL 与 CMSIS 厂商代码。
- `legged/Middlewares/`：FreeRTOS 和 USB 中间件。
- `legged/MDK-ARM/`：Keil 工程文件、链接脚本与目标配置。
- `legged/legged.ioc`：STM32CubeMX 配置源文件。

## 从仓库中识别出的工具链信息
- MCU：`STM32H723VGT6` / `STM32H723VGTx`。
- RTOS：FreeRTOS，使用 CMSIS-RTOS v2 包装层。
- STM32 固件包：`STM32Cube FW_H7 V1.12.1`。
- CubeMX 版本：`6.14.1`（来自 `legged/legged.ioc`）。
- Keil 工程：`legged/MDK-ARM/legged.uvprojx`。
- Keil 目标名：`legged`。
- Scatter 文件：`legged/MDK-ARM/legged/legged.sct`。
- `.ioc` 中虽有 `ProjectManager.CompilerLinker=GCC`，但仓库实际提交的是 MDK-ARM/ARMClang 工程，应以 `MDK-ARM` 工程为准。

## 构建命令
- 首选方式：在 Keil uVision 中打开 `legged/MDK-ARM/legged.uvprojx`，构建目标 `legged`。
- Windows + 已安装 Keil 时，推荐命令行构建：

```bash
UV4.exe -b legged/MDK-ARM/legged.uvprojx -t legged
```

- 若修改了头文件路径、生成文件或工程配置，优先执行完整重建而不是仅增量编译。
- 若修改了外设、引脚、中断、时钟、FreeRTOS、USB 等 Cube 配置，需要先从 `legged/legged.ioc` 重新生成代码，再进行构建。
- 重新生成后，必须检查自定义逻辑是否仍保留在 `/* USER CODE BEGIN */` / `/* USER CODE END */` 块中。

## 建议验证流程
- 先完成代码修改。
- 若涉及 Cube 配置或生成文件，先从 `legged/legged.ioc` 重新生成。
- 构建 Keil 目标 `legged`。
- 若修改了硬件交互逻辑，在 STM32H723 实板上验证并记录具体验证内容。
- 若修改了通信代码，重点检查包长、CRC、UART 收发、RS485 方向控制是否正确。
- 若修改了运动学或控制算法，重点检查单位、边界、奇异点、饱和限制及失败路径。

## 生成代码与第三方代码修改规则
- 优先修改项目自有代码：`legged/algorithm/`、`legged/driver/`、`legged/bsp/`。
- 修改 `legged/Core/` 和 `legged/USB_DEVICE/` 时要特别小心，这些目录很多文件由 Cube 自动生成，未来可能被覆盖。
- 对 Cube 生成文件，尽量只在 `/* USER CODE BEGIN ... */` 块内添加或修改自定义逻辑。
- 除非用户明确要求，不要修改 `legged/Drivers/` 和 `legged/Middlewares/` 中的厂商/中间件代码。
- 默认假设厂商目录未来可能被重新生成或整体替换。

## 头文件与依赖风格
- 若模块有自己的头文件，优先先包含自身头文件。
- 然后包含同层或近邻项目头文件，再包含 HAL/CMSIS 头文件，最后包含 C 标准库头文件。
- 保持与所在文件一致的包含写法；本仓库同时存在 `"foo.h"` 与 `<pid.h>` 两种风格。
- 不要增加未使用的头文件。
- 头文件依赖尽量局部化，避免在无关模块中引入过大的公共头。

## 代码格式风格
- 主要语言是 C，不是 C++。
- 老代码缩进风格略有混杂；新增代码默认使用 4 空格，除非局部代码强依赖原有风格。
- 在 Cube 生成文件中，函数和控制语句的大括号通常另起一行，尽量保持一致。
- `if`、`for`、`while` 后保留一个空格。
- 浮点字面量优先写成 `0.0f`、`1.0f`、`2.0f` 这样的形式。
- 宏和编译期常量使用全大写下划线命名。
- 语句保持直接清晰，不要为了“高级写法”压成晦涩的一行。
- 不要大规模格式化生成文件、厂商文件或与任务无关的旧代码。

## 类型与数据建模约定
- 协议字段、寄存器大小相关值优先使用 `<stdint.h>` 中的定长整数类型。
- 控制、运动学和电机空间计算主要使用 `float`，保持与现有实现一致。
- 有限状态、类型集合优先使用 `enum`，例如 PID 类型、腿 ID。
- 状态、数据包、控制器上下文优先用 `struct` 建模。
- 若已有 `#pragma pack`、`__attribute__((aligned(...)))` 等布局约束，必须保留。
- 不要静默扩大协议字段位宽，也不要调整已打包结构体的成员顺序。

## 命名约定
- 宏：全大写加下划线，例如 `VAL_LIMIT`、`SET_485_TR2`。
- 枚举值：全大写，通常带模块前缀，例如 `PID_ERROR_NONE`、`LEG_FL`。
- `typedef struct` / `typedef enum` 的命名风格在仓库内并不完全统一，如 `PID_Info_TypeDef`、`MotorParam`、`Leg`；修改时遵循本地文件风格即可。
- 函数命名既有驼峰也有下划线形式，优先保持所在文件现有模式。
- 仓库内存在全局腿对象和 HAL 句柄；除非确有硬件或 RTOS 需求，否则不要新增全局变量。

## 错误处理约定
- 遵循嵌入式项目现有风格：可恢复错误返回状态码，不可恢复初始化错误进入 `Error_Handler()`。
- 对数据包解析、数学计算和协议收发，优先做显式合法性检查并尽早返回。
- 关注 `NULL`、非法枚举值、除零、矩阵奇异、NaN、Inf 等异常情况。
- 面向执行器的命令要保留或增强限幅/饱和保护。
- 不要静默吞掉 HAL 或硬件接口失败；应返回 `HAL_StatusTypeDef`、设置错误状态或明确失败路径。

## 数值与控制代码注意事项
- 在代码与说明中尽量明确单位：弧度、弧度每秒、力矩、CRC 字节数、定时器 tick、UART 长度等。
- 除非已完整追踪协议或算法链路，否则不要随意修改现有缩放系数和映射常量。
- 电机命令在序列化前应先限幅。
- 修改运动学时，要特别注意左右腿符号约定、机身坐标系方向和角度定义。
- 对运动学改动，至少检查正运动学、逆运动学以及力/力矩映射之间是否仍一致。

## RTOS 与并发约定
- 当前 FreeRTOS 配置较轻，默认任务在 `legged/Core/Src/freertos.c` 中创建。
- 如果新增任务、队列、信号量、软件定时器，应明确其所有者、频率和生命周期。
- 不要在中断敏感或通信密集路径中引入无界阻塞。
- 区分 ISR 安全 API 与线程上下文 API，不要混用。
- 遵守 `legged/Core/Inc/FreeRTOSConfig.h` 中已配置的中断优先级限制。

## 代理修改代码时的工作准则
- 修改前先阅读目标文件，仓库风格并非完全一致。
- 优先做最小、可解释、局部的修改。
- 不要对生成代码、厂商代码做无关的大范围重排或美化。
- 如果修改的文件可能被 Cube 重新生成，要在回复中说明该修改是否具备“可再生成安全性”。
- 向生成文件中加逻辑时，优先放在最近的 `USER CODE` 块中。
- 如果变更可能要求重新生成 Cube 工程或重新构建 Keil 工程，要在结果中明确指出。
